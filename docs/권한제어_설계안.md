# Lead Agent 권한 제어 설계안

## 현재 상태

Lead 세션: `claude -p --dangerously-skip-permissions --allowedTools Bash,Read,Glob,Grep`

- `--dangerously-skip-permissions` → 모든 도구 호출이 무허가 실행
- `Bash` 도구 → 시스템 어디든 뭐든 실행 가능
- **기술적 제약 없음**, 프롬프트 지시에만 의존

Sub-agent도 동일: `--dangerously-skip-permissions` 사용 중

## 요구사항

- 프로젝트 디렉토리 내부: 자유로운 read/write
- 프로젝트 디렉토리 외부: read/ls 등 비파괴 작업만 허용
- 사용자가 GUI에서 allow/disallow 버튼으로 승인/거부 가능

## 구현 방안

### 방안 1: Claude Code Hooks (PreToolUse)

Claude Code는 도구 실행 전에 발동하는 `PreToolUse` 훅을 지원함.

```
흐름:
1. Lead가 Bash/Write/Edit 호출 시도
2. PreToolUse 훅 발동
3. 훅 스크립트가 .orchestra/permissions/request-{id}.json 작성
4. GUI의 fsnotify가 감지 → allow/disallow 다이얼로그 표시
5. 사용자 클릭 → .orchestra/permissions/response-{id}.json 작성
6. 훅 스크립트가 응답 파일 대기 → 읽어서 exit 0 (허용) 또는 exit 2 (차단)
7. Lead 계속 진행 또는 도구 실행 취소
```

**장점:** Claude Code 네이티브 기능 활용, 구조적으로 깔끔
**단점:** 훅 스크립트가 파일 폴링으로 응답 대기해야 함 (latency), 훅 설정 관리

### 방안 2: Bash 래퍼 스크립트

`claudestra` CLI를 통해서만 Bash 호출을 허용하는 래퍼:

```bash
#!/bin/bash
# allowed-bash.sh — claudestra 명령만 허용
if [[ "$1" == *"claudestra"* ]]; then
  eval "$@"
else
  echo "BLOCKED: claudestra CLI만 사용 가능합니다" >&2
  exit 1
fi
```

Lead의 `--allowedTools`에서 Bash 대신 이 래퍼를 사용.

**장점:** 간단, 확실한 차단
**단점:** Lead가 프로젝트 파일 직접 조작 불가 (Read/Glob/Grep으로만 읽기)

### 방안 3: allowedTools 세분화 + 프롬프트 강화

```
Lead: --allowedTools Read,Glob,Grep,Bash
                     ^^^^^^^^^^^^^^^^^ 읽기 전용
                                      ^^^^ claudestra CLI 전용 (프롬프트로 제한)
```

프롬프트에 강한 제약 추가:
```
[절대 규칙]
- Bash 도구는 오직 claudestra CLI 호출에만 사용하세요
- 파일 수정/삭제/이동은 반드시 claudestra assign을 통해 sub-agent에게 위임하세요
- 프로젝트 외부 파일은 절대 수정하지 마세요
```

**장점:** 구현 비용 0
**단점:** 프롬프트 기반이라 100% 보장 불가

### 방안 4: 하이브리드 (방안 1 + 3)

- 기본은 프롬프트 제약 (방안 3)
- Write/Edit/위험한 Bash 명령에만 PreToolUse 훅으로 승인 요청 (방안 1)
- Read/Glob/Grep/claudestra CLI는 자동 허용

```
PreToolUse 훅 로직:
  도구 == Read/Glob/Grep → 자동 허용
  도구 == Bash && 명령에 "claudestra" 포함 → 자동 허용
  도구 == Bash && 명령에 "ls/cat/find" 등 읽기 명령 → 자동 허용
  그 외 → GUI 승인 요청
```

## 결론: Docker 전까지는 방안 1 (PreToolUse 훅)이 필수

프롬프트만으로는 Lead agent가 예상 범위를 벗어나는 것을 막을 수 없음.
Docker 격리(Phase 4) 전까지 PreToolUse 훅 기반 승인 시스템이 반드시 필요함.

## 명령어 분류 체계

### 자동 허용 (Auto-allow)

훅이 즉시 `exit 0` 반환. 사용자 개입 없음.

| 분류 | 명령어 |
|------|--------|
| claudestra CLI | `claudestra *` (모든 서브커맨드) |
| 읽기 전용 | `ls`, `cat`, `head`, `tail`, `find`, `tree`, `wc`, `file`, `stat` |
| 검색 | `grep`, `rg`, `ag`, `fd` |
| Git 읽기 | `git status`, `git log`, `git diff`, `git branch`, `git show` |
| 도구 | Read, Glob, Grep (항상 허용) |

### 사용자 승인 필요 (Needs Approval)

훅이 `.orchestra/permissions/request-{id}.json` 작성 → GUI 다이얼로그 → 응답 대기.

| 분류 | 명령어 |
|------|--------|
| 파일 삭제 | `rm`, `rmdir`, `shred` |
| 네트워크 | `curl`, `wget`, `ssh`, `scp` |
| 패키지 관리 | `pip install`, `npm install`, `apt`, `brew` |
| Git 쓰기 | `git commit`, `git push`, `git checkout`, `git reset` |
| 프로세스 | `kill`, `pkill` |
| 시스템 | `chmod`, `chown`, `mount` |
| 쓰기 도구 | Write, Edit (프로젝트 외부 경로일 때) |

### 즉시 차단 (Instant Block)

훅이 즉시 `exit 2` 반환. 실행 불가.

| 분류 | 명령어/패턴 |
|------|-------------|
| 재귀 삭제 | `rm -rf /`, `rm -rf ~`, `rm -rf /*` |
| 외부 쓰기 | 프로젝트 디렉토리 외부로의 Write/Edit |
| 위험 명령 | `dd`, `mkfs`, `fdisk`, `:(){:|:&};:` |
| 권한 상승 | `sudo *`, `su *` |

## 아키텍처: PreToolUse 훅 → GUI 승인 플로우

```
1. Lead/Sub-agent가 도구 호출 시도
2. PreToolUse 훅 스크립트 실행
3. 명령어 분류:
   ├─ 자동 허용 → exit 0
   ├─ 즉시 차단 → exit 2 + stderr 메시지
   └─ 승인 필요 → 4단계로
4. 훅이 .orchestra/permissions/request-{uuid}.json 작성
   {
     "id": "uuid",
     "tool": "Bash",
     "command": "rm old-file.txt",
     "category": "파일 삭제",
     "agent": "developer",
     "timestamp": "2026-03-10T..."
   }
5. GUI의 fsnotify가 감지 → allow/disallow 다이얼로그 표시
6. 사용자 클릭 → .orchestra/permissions/response-{uuid}.json 작성
   { "id": "uuid", "allowed": true }
7. 훅 스크립트가 응답 파일 대기 (inotifywait 또는 폴링)
   → allowed=true: exit 0
   → allowed=false: exit 2
```

## 참고: Sub-agent 권한

Sub-agent는 이미 `--allowedTools`로 제한됨:
- Producer: `Read, Write, Edit, Glob, Grep, Bash` (자기 디렉토리 내)
- Consumer: `Read, Glob, Grep` (읽기 전용)

`cmd.Dir`로 작업 디렉토리가 격리되어 있지만, Bash로 `cd /` 등은 기술적으로 가능.
PreToolUse 훅은 Sub-agent에도 동일하게 적용되어야 함.
Docker 격리 (Phase 4)가 근본적 해결책.
